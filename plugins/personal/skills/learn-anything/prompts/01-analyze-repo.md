# Step 1: 分析 repo,提炼核心抽象

**用途**: 读完 repo 后,提炼出 3-7 个**学习价值最高**的核心抽象,作为后续章节规划的依据。

## 输入

- repo 的本地路径 (已 `gh repo clone`)
- README.md 内容
- 主要源文件结构

## 你要做的

1. **读 README**: 项目定位、核心卖点、入门示例
2. **扫源码结构**: 识别主模块/入口文件 (忽略 test/、docs/、examples/、node_modules 等)
3. **找核心抽象**: 哪些**概念/类/数据流**构成这个项目的灵魂? 一个新用户必须理解哪几个东西才算"懂"?
4. **画依赖**: 哪个抽象是基础,哪个建在其他之上

## 输出格式 (严格 JSON)

```json
{
  "repo": { "owner": "...", "name": "...", "url": "...", "oneliner": "一句话说这个项目干啥的" },
  "abstractions": [
    {
      "id": "kebab-id",
      "name": "中文名(可带英文原词)",
      "summary": "2-3 句话说清它是什么、解决什么问题",
      "evidence": [
        { "path": "src/foo.py", "lines": "10-45", "why": "这里定义了核心类 X" }
      ],
      "depends_on": ["其他 abstraction id"],
      "learn_priority": 1
    }
  ]
}
```

## 规则

- **3-7 个**抽象,少了不够学,多了学不动
- `learn_priority` 1 = 必须先学,越大越高阶
- `evidence` 每个必须指向**真实存在**的文件和行号,grep 得到
- 禁止列出显而易见的东西(比如"有 package.json")
- 禁止列出只在 examples/ 里出现的用法

## 自检

输出前自问:
- 去掉任何一个抽象,学习者还能懂这个项目吗? (如不能 → 保留)
- 这 N 个抽象之间的依赖关系画得出拓扑图吗? (画不出 → 太发散)
- evidence 里的文件路径都真实存在吗? (不是 → 删掉该条)

## 衔接下一步

输出 JSON 后**直接继续 Step 2**,不要停下问用户。可以简短告知用户提炼出了哪些抽象(一句话概述),然后立刻进入课程设计。
